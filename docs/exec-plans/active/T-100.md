# T-100 — occurrence 락 획득 순서 검증 + race condition 회귀 테스트

## 배경 및 목표

`BookingCommandService`는 capacity-affecting 연산(`createBooking`, `mutateBooking`) 수행 시
`occurrenceRepository.lock(occurrenceId)` 로 행 락을 먼저 획득하도록 설계되어 있다.
그러나 이 순서가 정적/동적으로 검증된 테스트가 없어 리팩터링 시 회귀 위험이 존재했다.

본 카드는 두 가지 테스트를 추가한다:
1. **단위 테스트** (`BookingCommandServiceLockOrderTest`): SpyOccurrenceRepository로 lock → getOrCreate 호출 순서 정적 검증
2. **통합 테스트** (`BookingConcurrencyAuditTest`): MySQL Testcontainer로 동시성 불변식 검증

## 결정 기록

1. **다중 occurrence 동시 락 AC 조정**: 카드 AC에 "다중 occurrence 시 ID 오름차순 락" 요구가 있으나,
   현재 BookingCommandService는 단일 occurrence만 락하므로 실제 다중 락 경로가 없다.
   정렬 정책 문서화 테스트로 범위를 좁힘.

2. **BookingConcurrencyAuditTest 패키지 위치**: 카드 WRITE 경로가 adapter/in/web/booking/이나
   해당 디렉터리가 없고, 유사한 MysqlBookingConcurrencyTest가 application/booking/에 있으므로
   동일 패키지에 배치.

3. **Docker 없는 환경**: BookingConcurrencyAuditTest는 disabledWithoutDocker=true
   (MysqlTestContainerSupport 상속)으로 Docker 미설치 시 자동 스킵.
