# T-004 — user 도메인 ApplicationService 도입 (계층 격리)

## 배경 및 목표

user 도메인에 전용 ApplicationService(UserService)가 없어 프로필 관리 로직이
auth 도메인의 MeService에 분산되어 있음. UserService를 도입하여 계층을 격리한다.

## 결정 사항

### D-1: UserPort vs 기존 UserRepository 공존
- `application/user/port/UserRepository.kt` 이 이미 존재하며 AuthCommandService, MeService, UserCommandService 등에서 사용 중
- 카드에서 요구하는 `UserPort` 인터페이스를 `application/user/UserPort.kt` 에 신규 생성
- `UserPort` 는 `findById/findByEmail/save/deleteById` 만 포함 (UserRepository 보다 slim)
- `UserService` 는 `UserPort` 에 의존 (새 Port)
- `JpaUserRepositoryAdapter`, `UserQueryAdapter` 는 `UserRepository` 와 `UserPort` 를 모두 구현
- `AuthCommandService` 등 기존 코드는 `UserRepository` 를 그대로 사용 — auth 코드 수정 금지

### D-2: displayName 검증 길이
- 카드에는 "255자 이하" 라고 기재되어 있으나 auth 도메인의 `requireValidDisplayName` 은 100자 제한
- UserService 내부 검증도 **100자** 로 통일 → 회원가입과 프로필 수정의 정책 일치

### D-3: MeService 처리
- `MeService.updateCurrentUser` → `UserService.updateProfile` 로 위임 (auth 의존성 제거)
- `MeService.getCurrentUser` → `UserService.getCurrentUser` 로 위임
- `MeService.getCurrentUserMemberships` 는 OrganizationQueryService 를 사용하므로 유지
- 결과적으로 MeController 는 profile 조회/수정을 UserService 로, memberships 는 MeService 로 위임

### D-4: 기존 UserCommandService 와 충돌 없음
- `application/user/UserCommandService.kt` 는 admin용 suspend/delete/restore 담당 — 역할 분리됨
- 신규 `UserService` 는 사용자 self-service 프로필 관리 담당
